01
Four things your pipeline still can't handle
Labs 04 and 05 ship files in Git to gateways. Four things were quietly left out of the repo, on purpose, until today.
  • Gap 1: secrets. The gateway's DB password, the API keys your deploy workflow scans with, the runner's PAT. They configure the system, but they can't be committed. Where do they live instead: locally, in Compose, in the gateway, in CI?
  • Gap 2: the database. Your Perspective screens read and write tables. The schema those screens depend on lives in a Docker volume: nothing versions it, nothing deploys it. Today it becomes files in the repo, shipped by the same pipeline.
  • Gap 3: third-party modules. MQTT drivers, charting modules, historians: .modl binaries the gateway needs at boot. Who versions them, how do they reach the gateway, and who accepts the license?
  • Gap 4: JAR files. JDBC drivers and Java libraries your scripts call. In 8.3 some are config resources, some sit on the gateway classpath: both have to ship with the deploy.
Builds on

Everything today extends the lab 04/05 stack: the local → test → production gateways, the .env.example convention, and the db-init/ folder you've already used without looking too closely at it.

Sorting is 80% of secrets management. Every value in your project is public, per-environment, or secret, and each box has a different home.

Public

Passwords, API keys, PATs, private keys, keystores. Anyone holding the value is you.

Part 1 · Secrets · the iron rule
03
A pushed secret is a burned secret
Lab 01 taught you commits are immutable snapshots. Here's the dark side of that superpower.
# Monday, 09:14: "just to make it work"
git commit -m "wire up historian"
+ IGNITION_API_KEY = "ak_live_9f3ce81b..."
# Monday, 09:40: "phew, removed it"
git commit -m "oops, remove key"
- IGNITION_API_KEY = "ak_live_9f3ce81b..."
# anyone, any clone, any time later:
git log -p | grep ak_live
+ IGNITION_API_KEY = "ak_live_9f3ce81b..."  ← still there
The only correct response

1. Rotate the credential, immediately. The old value is public property now; make it worthless.

2. Then, optionally, scrub history (git filter-repo, BFG). On a shared remote this is best-effort: forks, clones and CI caches keep copies.

Deleting the line and force-pushing without rotating is theatre.

Prevention beats response

Secret scanners (e.g. gitleaks) in scripts/validate.sh and CI catch the Monday-09:14 commit before it's pushed. You'll wire this in the assignment.

Step through the leak everyone commits once, and why "removing" it fixes nothing.
Part 1 · Secrets · the ladder
04
The secrets ladder: four rungs, from hardcoded to referenced
Each rung fixes the leak of the one below it. You don't need the top rung everywhere. You need to know which rung you're on.
0
Hardcoded in the repo plaintext in docker-compose.yaml or committed config
Leaks into: Git history, every clone, every fork. Never acceptable.
1
Env vars via .env + Compose interpolation what labs 02 to 05 already do
Out of Git. Still leaks into: docker inspect, compose config, the container's process env.
2
File-based secrets files mounted read-only at /run/secrets/<name>
Not in inspect, not in process env. A plaintext file on the host: the infra side owns and fills it.
3
Ignition 8.3 secret providers gateway config references a secret instead of containing it
Config becomes portable across local / test / production. Rungs 2 + 3 together are how our production gateways run. Next slides.
Climb the ladder rung by rung, bottom to top.
Part 1 · Secrets · rung 1
05
Rung 1: .env files, done properly
You've been on this rung since lab 02 without ceremony. Let's make the convention explicit: it's good, and it has a known ceiling.
  • .env is gitignored; .env.example is committed. The example documents every variable with a safe placeholder. Every lab repo ships one.
  • Compose interpolates ${VAR:-default} at parse time: safe defaults for the lab, real values from your .env.
  • The ceiling: the value still lands in the container's environment, visible in docker inspect, docker compose config, and to any code that can dump env.
# docker-compose.yaml: lab 04/05, unchanged
environment:
  GATEWAY_ADMIN_PASSWORD: "${GATEWAY_ADMIN_PASSWORD_LOCAL:-lab04password}"
  POSTGRES_PASSWORD: "${POSTGRES_PASSWORD:-ignition}"

# where the leak shows:
docker inspect lab06-timescaledb \
  | grep POSTGRES_PASSWORD        # plaintext
Verdict

Perfectly fine for a laptop lab. Not where you stop for shared test/production machines. That's rung 2.

Part 1 · Secrets · rung 2
06
Rung 2, file-based secrets: a file, not an env var
A secret is a file, not an env var. This one idea works everywhere we deploy: Docker Compose mounts it, Kubernetes mounts it the same way, and Ignition's secret provider can read it directly. It's how our production gateways run.
# docker-compose.yaml
secrets:
  postgres_password:
    file: ./secrets/postgres_password.txt   # gitignored

services:
  timescaledb:
    secrets: [postgres_password]
    environment:
      # official postgres image: _FILE suffix
      POSTGRES_PASSWORD_FILE: /run/secrets/postgres_password
  • Not in docker inspect, not in the process environment: the value only exists as a mounted file the process reads at startup.
  • Portable across platforms. Compose secrets:, Kubernetes secret volumes and plain bind mounts all end at the same place: a small read-only file at a known path.
  • Clean handover to infra. The repo commits the paths and names (plus *.example dummies); the infra team fills the real files per environment. Our production repo even ships committed dummy values for local test, so a fresh clone just works.
  • And CI can produce these files too: a deploy workflow can write GitHub secrets into password files on the target before compose up. Same mechanism end to end: slide 09.

Part 1 · Ignition 8.3 · secret providers
07
Embedded vs referenced secrets
Gateway config (DB connections, auth profiles, email…) needs credentials. 8.3 gives you two ways to store them, and the difference decides whether your config can travel.
Embedded secret
  • The value is encrypted by this gateway and stored inside its own config.
  • Fine for a single, hand-managed gateway: set it once in the UI, done.
  • The catch: the ciphertext is encrypted against that gateway's keys. A committed ciphertext only decrypts on the gateway that produced it: on a fresh container it's a broken reference.
  • For a pipeline that promotes the same config local → test → production: a dead end, unless you manage the keys yourself. That exception is slide 10.
Referenced secret
  • Config stores a reference: "get MSSQL_PASSWORD from provider PlantSecrets", never the value.
  • A file-type secret provider resolves it at runtime by reading a file: rungs 2 and 3 snap together at /run/secrets/<name>.
  • This is exactly our production setup: one file-type provider, ~16 named secrets, every DB connection and auth profile pointing at it by name.
  • Committed config contains no secret material, so it can be diffed, reviewed and deployed anywhere.
Rule of thumb

Embedded = pet gateway. Referenced = pipeline. If a gateway's config is in Git (and after labs 04 and 05, yours is), its secrets should be referenced.

08
One config, three gateways, three values
The same committed config works everywhere, because each environment injects its own value under the same secret name. Config stays in Git; values stay out.
📄 database-connection config.json, committed (real shape, from our production repo)
"password": {
  "type": "Referenced",
  "data": {
    "providerName": "PlantSecrets",
    "secretName": "MSSQL_PASSWORD"   ← a name,
                                    not a value
  }
}
→
local resolves MSSQL_PASSWORD → /run/secrets/MSSQL_PASSWORD = a committed dummy value, so a fresh clone just works
test resolves MSSQL_PASSWORD → the file infra filled on the test host
production resolves MSSQL_PASSWORD → the production value, which only production ever holds
Same pattern, everywhere

You already use this model in CI: one workflow file, and the lab-gateway-test / lab-gateway-production GitHub environments each supply their own IGNITION_API_KEY. Referenced secrets bring the identical idea inside the gateway.

Part 1 · Secrets · in the pipeline
09
Secrets in GitHub Actions, and turning them into secret files on deploy
Lab 04 made you create environment secrets. Two things to add: the discipline that goes with them, and the move that connects them to rung 2: the deploy workflow writes GitHub secrets into the password files the gateway's provider reads.
  • Environment-scoped, not repo-wide. lab-gateway-production holds the production values; a PR build against test can't even see them. Add required reviewers on the production environment for a human gate.
  • GitHub secrets → secret files. The deploy job materializes each secret as a file on the target host, with tight permissions before the value lands, then compose up mounts them at /run/secrets/. One source of truth in GitHub, files on the box, references in the gateway: the whole ladder in one pipeline run.
  • Masking is a seatbelt, not a licence. Actions replaces known secret values with *** in logs, but don't echo them, don't set -x around them, don't write them into artifacts.
  • Forks don't get secrets. pull_request runs from forks see empty secrets by design: one more reason deploys trigger on push/tag, never on PR.
# deploy.yml: write secrets to files, then deploy
jobs:
  deploy:
    runs-on: [self-hosted, lab06]
    environment: lab-gateway-test   # ← scoping
    steps:
      - name: Materialize secret files
        env:
          POSTGRES_PASSWORD: ${{ secrets.POSTGRES_PASSWORD }}
        run: |
          umask 177                    # files land 0600
          mkdir -p secrets
          printf '%s' "$POSTGRES_PASSWORD" \
            > secrets/postgres_password.txt
      - name: Deploy
        run: docker compose up -d
Never

run: echo ${{ secrets.X }}, and never interpolate a secret straight into a shell command line; pass it through env: and write with printf, so it stays out of the run log. Our production boot script does exactly this: umask + chmod 0600 before the value is written.

Part 1 · Secrets · the list
10
The never-commit list for an Ignition repo
Print this. Tape it to the monitor of whoever owns the repo.
  • .env, any real one. (.env.example with placeholders: yes, please.)
  • secrets/ contents, where only the *.example files are tracked.
  • PATs & API keys: ghp_…, runner registration tokens, gateway API keys.
  • Gateway backups (.gwbk) from real systems. They carry users, DB connections and keystores in one convenient bundle.
  • Keystores / TLS private keys: ssl.pfx, metro-keystore, *.p12.
  • The one defensible exception: ciphertexts, if you own the keys. 8.3 ships a secrets-management key CLI tool (ignition-secrets-tool.sh) that manages the gateway's root key and key-encryption keys as JSON files under data/config/ignition/keys/. Distribute the same keys to your gateways (the way redundant pairs already do) and committed ciphertext decrypts everywhere: an alternative to file-based secrets. The keys themselves become the secret: infra distributes them, and they never, ever go in Git. Docs: Secrets Management Key CLI Tool.
Enforcement, not vigilance

Humans forget; hooks and CI don't. This repo's scripts/validate.sh runs a secret scan (gitleaks) and ci.yml repeats it on every PR: a leaked key fails the build before a reviewer ever sees it.

# .gitignore: the relevant lines
.env
secrets/*
!secrets/*.example
*.gwbk

# validate.sh (excerpt)
gitleaks detect --source . # zero findings = green

Part 2 · Database · the problem
11
Schema drift: the failure mode
The deploy pipeline from lab 04 ships the screen. Nothing ships the table the screen needs. That gap has a schedule: it bites on release day.
📄 what Git deploys to production
Perspective view "DowntimeLog"
  ↳ named query: SELECT reason, minutes
    FROM downtime_log ...   ← new screen,
                              shipped ✅
🗄 what the production database has
ignition_production=# \dt
  batch_log      ✓
  recipe_params  ✓
  downtime_log   ✗ created by hand
                   on test only, three
                   weeks ago
  • "Worked on test." Because someone manually created the table on test. Production was never told. The screen 500s on release night.
  • No history. Who changed the column type? When? Why? With hand-run SQL there is no equivalent of git log for your schema.
  • The fix is the same fix as always: make schema change files in the repo, reviewed in the same PR as the screens that need them, applied by the pipeline.

Part 2 · Database · what you already have
12
db-init/: bootstrap, not deployment
Labs 04/05 already mount SQL into the database container. Time to be precise about what that mechanism can and cannot do.
# docker-compose.yaml: labs 04/05, unchanged
volumes:
  - ./db-init:/docker-entrypoint-initdb.d:ro

# db-init/01-create-env-databases.sql
CREATE DATABASE ignition_test;
CREATE DATABASE ignition_production;
The one rule

Postgres runs /docker-entrypoint-initdb.d once, on first init of an empty data volume. Volume already exists? Nothing runs. Ever again.

  • Great for: creating the databases, the app user, extensions, the "day zero" shape of a fresh environment.
  • Useless for: changing a database that already has data. Editing db-init/*.sql after first boot changes nothing on any running environment.
  • The trap: "I added the table to db-init and it worked!", because you also wiped your volume. Production doesn't get its volume wiped.
  • So: day-zero shape → db-init/. Day-one-and-beyond change → migrations →

Part 2 · Database · migrations
13
Migrations: numbered SQL files, applied in order, tracked in the database
We use golang-migrate, the same tool our file-based production repo uses. Files come in pairs (.up.sql / .down.sql), numbered; migrate up applies the ones a database hasn't seen yet, in order, and records where it got to.
📄 db-migration/migrate/0001_baseline.up.sql (+ .down.sql)
CREATE TABLE batch_log (…);
CREATE TABLE recipe_params (…);
📄 db-migration/migrate/0002_add_downtime_log.up.sql (+ .down.sql)
CREATE TABLE downtime_log (
  id BIGSERIAL PRIMARY KEY,
  reason TEXT, minutes INT, …);
📄 db-migration/migrate/0003_downtime_add_area.up.sql (+ .down.sql)
ALTER TABLE downtime_log
  ADD COLUMN area TEXT;
migrate up
→
applies what's
missing, in order
schema_migrations, inside each database
0001baseline · applied on test and production
0002add downtime log · applied on test and production
0003downtime add area · pending on production, applies on next deploy
The rules that make this work (straight from the production repo's docs/MIGRATIONS.md)

Migrations always come in pairs (up applies, down undoes) · never edit a migration after it has deployed anywhere: fix mistakes with a new numbered migration · the tool tracks each database's position in schema_migrations, so every environment catches up at its own pace. It's the Git mental model, applied to your schema.

The deploy pipeline, now with a database step
Lab 04's five steps get a sixth, and the order is the whole point: the schema must be ready before the gateway scans in screens that need it.
1

→
3 · new

migrate up against the target env's database. Fails → run stops here.

→
4

→
5
Scan

POST /scan: gateway picks up projects + config, no restart.

→
6
Verify

Smoke check: gateway up, query the new table.

Why migrate before ship

Old screens tolerate a new table; new screens don't tolerate a missing one. Expand-first ordering means a half-failed run leaves production on old screens + new (unused) schema, a safe place to stop.

On failure

The migrate step failing fails the whole run: no screens ship. Fix forward with a new migration and redeploy. Worth knowing: our file-based production pipeline runs migrations after the scan, with continueOnError. Look at it in the we-do and spot what can go wrong with that ordering.

Part 3 · third-party modules
15
Third-party modules: commit the .modl, load it from a folder, accept the license in config
The .modl binary lives in Git, a JVM flag points the gateway at the folder, and license acceptance is data, not a click. You never type the fingerprint by hand: the gateway computes it.
The pattern (both production repos, identically)
  • Commit the binary. Drop the .modl into third-party-modules/ — it's tracked in Git like any other file (Git LFS in the real repos; these ship small enough to commit directly).
  • The folder is already wired. Every gateway's compose command: passes -Dignition.gateway.externalModulesFolder=/third-party-modules — the container path the repo folder is mounted at. Nothing to change; that flag is what makes the gateway load from the folder at boot instead of the UI.
  • Ship it like everything else. File-based: the pipeline copies the folder to the gateway host. Image-based: one COPY layer, like the production Dockerfile yesterday.
Where do certFingerprint & licenseAgreementHash come from?

You don't guess them. List the module in the three env vars, add a minimal manifest entry, boot once — the gateway accepts headlessly and writes both values back into modules.json. Commit what it wrote. Now a fresh gateway with no env vars accepts from config alone.

# 1 · compose env vars — list the module id in all three
GATEWAY_MODULES_ENABLED:  …, com.mussonindustrial.embr.charts
ACCEPT_MODULE_LICENSES:   …, com.mussonindustrial.embr.charts
ACCEPT_MODULE_CERTS:      …, com.mussonindustrial.embr.charts

# 2 · modules.json — you write only these two lines:
"com.mussonindustrial.embr.charts": {
  "filename": "/third-party-modules/Embr-Charts-Ignition83-4.0.1.modl",
  "onStartup": "enabled"
}

# 3 · boot once → the gateway ADDS these itself:
  "certFingerprint": "e5a3cf3f06627c175b68b0122ac8f2c3f9c992e2",
  "licenseAgreementHash": 3266212556
# commit them → fresh gateways accept headlessly, every boot

Part 3 · JAR files
16
JAR files: JDBC drivers are config resources, other libraries go on the classpath
Two different kinds of JAR, two different homes. Both are versioned in the repo and shipped by the pipeline: nobody copies a JAR onto a server by hand.
Kind 1 · JDBC drivers: part of the 8.3 config tree
  • In 8.3 a database driver is a config resource: the JAR sits inside the resource folder, next to its config.json.
  • Committed and deployed in production: config/resources/core/ignition/database-driver/PostgreSQL/postgresql-42.7.2.jar, with resource.json listing the JAR in its "files".
  • It ships with the ordinary config copy. No special handling: deploy the config tree, and the driver arrives with it.
Kind 2 · library JARs: on the gateway classpath
  • Java libraries your gateway scripts call. Our image-based production repo ships amqp-client (RabbitMQ) and an SMB client, with their pinned dependency JARs alongside.
  • They live in modules/jar/ in the repo, with a README that records where each JAR came from (the Maven Central command), and get copied to lib/core/gateway/.
  • Pin the versions, commit the binaries. The gateway has no package manager: your repo is the lockfile.
One rule covers all of Part 3

Modules and JARs are deployable state, exactly like a view or a config file. If the gateway needs it to run, it lives in the repo, it travels through the pipeline, and a fresh gateway comes up complete with no hands on it.
