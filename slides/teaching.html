Why more than one gateway?

The invariant

The image is built exactly once — on the push to develop. Every arrow after that is docker buildx imagetools create: a server-side re-tag. The digest on prod is byte-identical to what dev ran. Rebuild anywhere along the way and you've thrown your testing evidence in the bin.

Two ways teams do this — one is a trap
Branch-per-environment feels natural in git. It's the most common way multi-environment Ignition projects go wrong.
Branch-per-environmentBuild-once / promote-many (this lab)
Deploy =merge dev → test → prod branchesre-tag one immutable image
Prod runsa fresh build of the prod branch — not the bytes you testedthe exact digest test validated

Say it once, clearly

Git history describes the product. Registry tags describe where each build stands. The moment environments become branches, those two get tangled and every deploy is a merge with side effects.

Promotion · gates
05
Gates: what stands between stages
Each hop has a different kind of gate — and the prod gate needs zero workflow changes.
HopGateMechanism
feature → developreview + CI greenPR + branch protection
develop → devnone — automaticdeploy.yml on push
dev → testa human says "freeze"pushing release/*
test → prodacceptance + approvaltag v* + required reviewers
  • Workflows say what happens. Environments say what's allowed to happen, and by whom. Required reviewers on lab-gateway-prod pause the deploy job until someone clicks approve.
  • Gates you'd add in real life: smoke tests against the freshly deployed stage, change-window checks, a digest-equality assertion before prod.
  • The release branch is a gate too — a human decision ("this is what we'll test") expressed as a git action, so it's audited like everything else.
In the lab

You'll add yourself as a required reviewer on lab-gateway-prod, tag a release, and watch the prod deploy sit and wait for you. Approval is a button — and an audit-log entry.

Configuration · inventory
06
What actually differs per environment?
Take an inventory before choosing a mechanism. Most of the gateway is — and must stay — identical.
Identical everywhere (or you're not testing prod)
  • Projects — views, scripts, styles, session props
  • Tag structures & UDT definitions
  • The module set (modules.json) and module versions
  • Historian / tag-provider structure
  • Ignition version, JVM, base image
Legitimately different per environment
  • Database connectURL (dev DB vs prod DB)
  • OPC UA endpoints / device addresses (sim vs plant)
  • Gateway name, gateway-network targets
  • Alarm notification targets (test SMTP vs real pagers)
  • Credentials & API keys — secret, special handling
The rule

If it differs per environment, it must be parameterized and versioned — never hand-edited on the gateway after deploy. A hand-edit is a difference that survives exactly until the next deploy erases it. That is the 2 a.m. incident.

  • All scopes ship in the artifact. Dev's image contains prod's overlay and vice versa — selection, not content, differs per gateway.

Silent fallback is the sharp edge

A missing overlay doesn't error — it inherits. The test gateway will happily boot and write test data into whatever core points at. In the lab, you create tst/ and ship it through the pipeline to fix exactly this.

Configuration · the framework
09
Every knob goes in one of three buckets
This classification is configuration management. When in doubt, argue about the bucket — not the mechanism.
Bucket 1

Same everywhere

Identical in every environment — differences here mean you're not testing what you ship.

  • historian provider settings
  • tag providers, IdP config
  • session props, module set
→ core/ · in git · in the image
Bucket 2

Per-env, readable

Differs by design, safe for anyone with repo access to read.

  • DB connectURL
  • OPC UA endpoints, device IPs
  • alarm SMTP host, gateway name
→ scope overlay · in git · in the image
Bucket 3

Secret

Differs per env and must not be readable. Currently: encrypted blobs sitting in git. Not good enough.

  • DB passwords
  • API keys, PATs
  • cloud credentials
→ NOT in git → Lab 07 (this afternoon)
Sharp edges

One resource can straddle buckets: an IdP config is bucket 1 with a bucket-3 client secret inside it. And modules.json is bucket 1 in spirit but lives outside the scope tree. The drill in the lab makes you defend ten of these classifications.

Configuration · layers
10
Bake-time · boot-time · run-time
Buckets say where a value lives. Layers say when it takes effect — and what it costs to change.
LayerSet whereChanging it meansExamples
Bake-time Dockerfile / image content new image build + full promotion projects, .modl binaries, all scope overlays
Boot-time compose env vars + JVM args recreate the container -Dignition.config.mode, admin creds, memory, ports, GATEWAY_MODULES_ENABLED
Run-time gateway APIs / scan nothing restarts project resources via file-based scan — Lab 04's inner loop
The multi-gateway insight

Overlays are bake-time content, but their selection is boot-time (mode=…). That's the whole trick: one image carries every environment, and a one-line boot flag decides which personality it wakes up with.

Fan-out · matrix
11
Fan-out with a matrix
GitHub Actions runs the same deploy job once per gateway. Promotion logic doesn't change — only the target list grows.
deploy:
  needs: promote
  strategy:
    matrix:
      gateway: [prod, prod-b]   # the fleet
    max-parallel: 1             # one at a time → canary
    fail-fast: true             # a failure HALTS the rollout
  runs-on: [self-hosted, lab06]
  # real fleets: one runner label per site,
  # runs-on: [self-hosted, "${{ matrix.gateway }}"]
  environment: lab-gateway-${{ matrix.gateway }}
  steps:
    - run: scripts/deploy-image.sh ${{ matrix.gateway }} $TAG
  • One matrix entry = one gateway. Add a line to add a site. The artifact, the tag, and the approval gate are untouched.
  • max-parallel: 1 is your canary. Order the list least-critical first; a failed healthcheck stops the rollout before it reaches line A.
  • Per-site runners. In real fleets each site runs its own self-hosted runner with a site label — the matrix routes each deploy to the right building.
Where per-site differences go

Line B's device IPs are not a different image and not a workflow variable — they're a scope overlay, same as dev vs prod. Horizontal differences use the same bucket framework as vertical ones.

Fan-out · failure modes
12
Fleets fail partially — plan for it
The steady state during any rollout is a mixed-version fleet. The question isn't how to avoid it, it's how to see it and get out of it.
  • Drift must be visible in seconds. "Which version is line B running?" = docker inspect -f '{{ index .RepoDigests 0 }}' per gateway — or better, a status view that shows every gateway's version. Never archaeology.
  • Health-gate every hop. Deploy → wait for /StatusPing → only then take the next gateway. A failed check must halt the rollout, not skip the box.
  • Rollback = deploy with an older tag. Same workflow, same mechanics, workflow_dispatch with v1.0.0. If rollback has different mechanics than deploy, it will fail exactly when you need it most.
  • Design the app for mixed versions. Alarms, gateway-network calls, shared DB schemas — all briefly span two versions during a rollout. (DB schema migrations: that was Day 3; secrets rotation: this afternoon.)
Contrast worth naming

Ignition's own EAM / gateway network can push projects between gateways too — a pull-from-inside model. It solves delivery, not promotion: no build-once guarantee, no gates, no audit trail in git. Fine for some fleets; know what you're giving up.

The whole model on one slide
If you remember three sentences from this hour, make it these.
Promotion

Build once, re-tag forward

One image built on develop. release/* promotes it to test; an approved tag promotes test's image to prod. The digest never changes; the trust does.

Configuration

Content baked, selection booted

Every environment's overlay ships in the artifact. -Dignition.config.mode picks one at boot. Differences are versioned; nothing is hand-edited.

Fleet

Horizontal = more tags on the same move

A matrix fans the identical release out; health gates order it; rollback is the same workflow with yesterday's tag.

Still unsolved — on purpose

Every password in those overlays is an encrypted blob committed to git. This afternoon (Lab 07) we rip those out and do secrets properly. Hold that itch.
