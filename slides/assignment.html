Warm-up · together
WARM-UP · 1
Triage every secret-shaped value in the repo
Sorting is 80% of secrets management: public, per-environment, or secret. Do the sort on your own clone.
1
Grep the repo for candidate secrets (see command) and skim the hits.
2
Sort each hit into public / per-environment / secret in NOTES.local.md. Defaults like ${POSTGRES_PASSWORD:-ignition} count: which box is the default value in?
3
Check the fence: git check-ignore .env secrets/anything must both hit; *.example files must be tracked.
4
Run scripts/validate.sh: the secret scan must report zero findings. That's your baseline for the rest of the hour.
# step 1: hunt for secret-shaped values
grep -rniE "password|api_key|token|secret" \
  --exclude-dir=.git .

# step 3: is the fence up?
git check-ignore .env secrets/postgres_password.txt
git ls-files secrets/   # only *.example should list
Three boxes

Public → committed plain. Per-environment → committed defaults + env vars. Secret → never committed. Every value in every repo you'll ever touch goes in exactly one.

Warm-up · together
WARM-UP · 2
Leak a fake key, "remove" it, read it back from history
Prove to yourself that deleting a committed secret fixes nothing, and write down the real incident response.
1
On a scratch branch drill/leak, commit a file with an obviously fake key: API_KEY="ak_live_0badc0ffee".
2
"Fix" it: delete the line, commit again.
3
git log -p | grep ak_live: still there, in every clone, forever.
4
Write the two-line incident response in NOTES.local.md: 1. rotate the credential, 2. then (optionally) scrub history. In that order, always.
5
Delete the branch. Never push it.
git checkout -b drill/leak
echo 'API_KEY="ak_live_0badc0ffee"' > oops.txt
git add oops.txt && git commit -m "wire up historian"

# the "fix"
git rm oops.txt && git commit -m "remove key"

# the lesson
git log -p | grep ak_live
+API_KEY="ak_live_0badc0ffee"   ← still readable

git checkout main && git branch -D drill/leak
Iron rule

A pushed secret is a burned secret. Rotation makes it worthless; deletion only hides it from people who don't know git log -p.

Solo · Part 1 · Secrets
Part 1 · solo · ±20 min · exercises/lab.md
Move the Postgres password up the ladder, rung by rung

One credential, three moves, ending exactly where the Wilms production setup lives: a secret file on disk, a referenced secret in the gateway, and CI as the source of the value.

  • 1A: compose environment: → a secret file at /run/secrets/.
  • 1B: the gateway DB connection points at a referenced secret resolved from that file.
  • 1C: the deploy workflow writes the secret file from a GitHub secret, so dev's value never lives in the repo.
Mustry Academy · CI/CD for IgnitionPart 1 · A → B → C
Part 1 · A
1A
Turn the Postgres password into a secret file
From an env var anyone can docker inspect, to a read-only file only the process reads.
1
See the leak first: docker inspect lab06-timescaledb | grep POSTGRES_PASSWORD → plaintext.
2
Create the file from the committed example: cp secrets/postgres_password.txt.example secrets/postgres_password.txt. Keep the same value: Postgres only sets the password on first volume init.
3
Edit docker-compose.yaml: top-level secrets: block, attach to the DB service, swap to POSTGRES_PASSWORD_FILE.
4
Recreate and prove it: inspect shows no password; the file is at /run/secrets/postgres_password inside the container.
# docker-compose.yaml
secrets:
  postgres_password:
    file: ./secrets/postgres_password.txt

services:
  timescaledb:
    secrets: [postgres_password]
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/postgres_password

# step 4: the proof
docker compose up -d timescaledb
docker inspect lab06-timescaledb | grep POSTGRES
docker exec lab06-timescaledb \
  ls -l /run/secrets/
Same value

Changing the password here won't change it in an existing database volume. Ladder drills change where the value lives, not the value.

Mustry Academy · CI/CD for IgnitionPart 1 · A of C
Part 1 · B-C
1B-1C
Reference the secret in the gateway, then let the pipeline write the file
1B closes the gateway side: config holds a name, not a value. 1C closes the CI side: GitHub is where the real value lives.
1
1B. On the local gateway, create a file-type secret provider and add a secret named POSTGRES_PASSWORD whose file path is the mounted secret file.
2
Re-point the DB connection's password at the referenced secret. Test: the connection stays Valid.
3
Prove nothing leaked: grep the exported config for the password value → no hits. The reference by name is all that's there (the "type": "Referenced" shape from the teaching).
4
1C. In your fork: add POSTGRES_PASSWORD as an environment secret, then add the Materialize secret files step to deploy.yml (before compose up).
5
Push through a PR and watch the run: the workflow writes the file with umask 177 + printf, and the log shows no value anywhere.
# deploy.yml: the step you add in 1C
- name: Materialize secret files
  env:
    POSTGRES_PASSWORD: ${{ secrets.POSTGRES_PASSWORD }}
  run: |
    umask 177
    mkdir -p secrets
    printf '%s' "$POSTGRES_PASSWORD" \
      > secrets/postgres_password.txt

# 1B step 3: prove the export is clean
grep -r "$(cat secrets/postgres_password.txt)" \
  services/config/ || echo "clean ✅"
✅ Part 1 gate

scripts/validate.sh green, secret scan zero findings, and you can narrate the full path: GitHub secret → file on the host → /run/secrets → provider → reference in config.

Solo · Part 2 · Database
Part 2 · solo · ±20 min · exercises/lab.md
Ship a schema change as a migration the pipeline applies

db-init/ is bootstrap: it never touches an existing database. A schema change ships as a numbered golang-migrate pair, applied by deploy.yml before the screens that need it.

  • 2A: write 0002_add_downtime_log as an up/down pair, apply it locally, read schema_migrations.
  • 2B: add the migrate step to deploy.yml before the ship step, and watch the pipeline migrate dev.
Mustry Academy · CI/CD for IgnitionPart 2 · A → B
Part 2 · A
2A
Write migration 0002 as an up/down pair, apply it locally
Golang-migrate conventions, exactly like the Proferro repo: 4-digit number, descriptive name, always a pair.
1
Create db-migration/migrate/0002_add_downtime_log.up.sql and …0002_add_downtime_log.down.sql. The down undoes exactly what the up does.
2
Apply it locally: scripts/migrate.sh up (golang-migrate in Docker, like Proferro's migrate.sh).
3
Read the ledger: the schema_migrations table now says version 2, not dirty. Run migrate.sh up again → no change: it only applies what's missing.
4
Now the rule the tool won't enforce for you: edit an already-applied migration file. Nothing complains, and that's the trap: other environments would silently diverge. Undo the edit; changes go in 0003. That's why Proferro's docs/MIGRATIONS.md makes it a written rule.
# 0002_add_downtime_log.up.sql
CREATE TABLE downtime_log (
  id BIGSERIAL PRIMARY KEY,
  reason TEXT NOT NULL,
  minutes INT NOT NULL,
  area TEXT
);

# 0002_add_downtime_log.down.sql
DROP TABLE downtime_log;

# apply + inspect the ledger
scripts/migrate.sh up
docker exec lab06-timescaledb psql -U ignition \
  -d ignition_loc -c 'SELECT * FROM schema_migrations;'
Pairs, always

"All migrations must come in pairs" and "never edit a deployed migration" are the two Proferro rules that keep every environment replayable from zero. The tool tracks position; the discipline is yours.

Mustry Academy · CI/CD for IgnitionPart 2 · A of B
Part 2 · B
2B
Add the migrate step to deploy.yml, before the ship step
Old screens tolerate a new table; new screens don't tolerate a missing one. So: migrate first, and a migrate failure stops the whole run.
1
Add the Migrate database step to .github/workflows/deploy.yml, before the ship-files step, targeting the dev database. The DB password comes from the secret file Part 1 set up.
2
Branch, add a small screen change that reads downtime_log, and open a PR with both the migration and the screen: schema and screens travel in the same PR.
3
Merge and watch the run: validate → migrate dev DB → ship files → scan → verify.
4
Prove it was the pipeline, not you: query dev's schema_migrations: version 2, applied during the run.
# deploy.yml: the step you add
- name: Migrate database
  run: scripts/migrate.sh up --database ignition_dev
# placed BEFORE the ship/scan steps, and NOT
# continueOnError: a failed migration stops the run.

# step 4: the proof, against the dev database
docker exec lab06-timescaledb psql -U ignition \
  -d ignition_dev -c 'SELECT * FROM schema_migrations;'
✅ Part 2 gate

A green deploy run whose log shows migrate before ship, and dev's ledger at version 2. Bonus debrief point: Proferro runs this step after the scan with continueOnError: true. What can go wrong there that can't go wrong in yours?

The binaries become deployable state: committed, license-accepted in config, shipped by the same run as everything else. The proof of success is a gateway that boots complete with no hands on it.

Mustry Academy · CI/CD for IgnitionPart 3 · A → B
Part 3 · A
3A
Add a third-party module with headless license acceptance
The module loads from the external-modules folder, and the license is accepted by config, not by a click.
1
Pick a module from third-party-modules/ that isn't enabled yet (the repo ships spares). Confirm the compose command: passes -Dignition.gateway.externalModulesFolder.
2
Enable it in services/modules.json: "onStartup": "enabled" plus the certFingerprint and licenseAgreementHash from the exercise file.
3
Ship it through the pipeline (PR → merge → deploy run).
4
Verify on dev: Config → Modules shows it Running. Nobody clicked an install dialog: that's the point.
5
Negative test: comment out the licenseAgreementHash, redeploy, and see what a module without accepted terms does on a fresh boot. Put it back.
# services/modules.json: what you add
"com.example.vendor.module": {
  "filename": "/third-party-modules/Vendor-signed.modl",
  "onStartup": "enabled",
  "certFingerprint": "…from exercises/lab.md…",
  "licenseAgreementHash": 1007003261
}
Fresh-boot test

The real proof is a fresh gateway coming up complete. If your dev gateway is file-based, ask: what would this look like in yesterday's image-based deploy? (Answer: one COPY layer and the same manifest.)

3B
Find the JDBC driver JAR in the config tree
In 8.3 a database driver is a config resource: the JAR travels with the ordinary config deploy. Prove it to yourself.
1
Find the driver: find services/config -name "*.jar" → the PostgreSQL driver sits under …/database-driver/PostgreSQL/, next to its config.json.
2
Open the sibling resource.json: the JAR is listed in "files". It's a versioned resource like any view or connection.
3
Answer in NOTES.local.md: how does this JAR reach the dev gateway? (Hint: the same ship step as everything else. No special JAR handling exists, and none is needed.)
4
Compare with the other kind: where do Wilms' library JARs (RabbitMQ client) land, and why can't those live in the config tree?

3
Secret scanner in CI. Add a gitleaks job to ci.yml (with fetch-depth: 0: the scanner must see history, not the tip). Test it with a fake-key PR → red ❌ → close without merging.
# stretch 3: ci.yml secret-scan job
secret-scan:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0
    - uses: gitleaks/gitleaks-action@v2
Two philosophies

Stretch 1 and Part 1 are competing answers to the same problem: references + files, or ciphertexts + shared keys. Both are defensible; know what each makes easy and what each makes dangerous.
